---
name: Ví chợ (market wallet) design
description: How the wallet-funded virtual stock system for chợ-sourced tenant products works — balance currency, gating, debit timing, deposit matching.
---

- Ví chợ balance (`AdminUser.market_wallet_balance`) is always VND-denominated, matching `Product.source_price`'s currency — even though top-ups only happen via crypto (USDT). Deposits store both the raw crypto `amount`/`currency` (for on-chain matching) AND a locked-in `vnd_credit_amount` (computed from the exchange rate at request time) that's what actually gets credited to the wallet on confirmation. Never credit the raw USDT amount into the VND balance directly.
- Virtual stock formula: split a tenant's wallet balance evenly across their attached `source_type=api` products (`budget_per_product = balance / count`), then `floor(budget_per_product / product.source_price)` per product. Computed live, never cached. Combined with real supplier stock via `min()` — wallet budget caps stock, never invents phantom stock.
- Owner's own products/tenant are never wallet-gated (owner IS the real supplier). Owner's `market_wallet_balance` row instead tracks how much they've prepaid to the real upstream supplier — same column, different meaning depending on `is_owner`.
- Sale debit (cost + 2% platform fee) happens in ONE atomic combined ledger row at order `completed` transition, guarded by `Order.market_wallet_debited` boolean to prevent double-debit. Primary blocking mechanism against overselling is `get_best_source` returning `None` when virtual stock is 0 (routes into existing `paid_waiting_stock` flow) — the debit-at-completion hook only races defensively and never reverses an already-delivered sale.
- Market wallet deposits reuse the exact same on-chain scanning loops as customer wallet deposits (same owner-controlled wallet address), via a parallel `_get_pending_market_deposits`/`_process_crypto_tx_for_market_deposits` path in `services/crypto_monitor.py`, with its own `CryptoTransaction.matched_market_deposit_id` column (distinct from `matched_deposit_id`) so one on-chain tx can't double-fund both a customer wallet and a tenant's ví chợ.
- Cross-tenant catalog sharing (tenant browsing owner's `ApiConnection`/`ApiProduct` catalog directly) is explicitly NOT implemented — ví chợ only gates a tenant's own already-attached `source_type=api` products. See follow-up task on shared chợ catalog.
